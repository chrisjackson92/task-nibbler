## Findings

### Finding #1 — CRITICAL: `PreRegister` returns a UUID that does not match the DB row (BLOCKING)

**File:** `internal/services/attachment_service.go` — `PreRegister()`, lines 131–157

**Root cause:**

```go
// 1. Service generates its own UUID for the S3 key path
attachmentID := uuid.New()           // ← service-generated UUID
s3Key := fmt.Sprintf("%s/%s/%s%s",
    userID.String(), taskID.String(), attachmentID.String(), ext)

// 2. S3 presign uses the service-generated UUID
uploadURL, expiresAt, err := s.s3.PresignPutURL(ctx, s3Key, ...)

// 3. DB row is created — DB assigns its OWN gen_random_uuid()
//    The returned *Attachment (with DB-generated UUID) is DISCARDED
_, err = s.repo.Create(ctx, taskID, userID, s3Key, ...)
//     ↑ repo INSERT has no `id` column → DB generates different UUID

// 4. Response returns the SERVICE-generated UUID — NOT the DB row's UUID
return &PreRegisterResponse{
    AttachmentID: attachmentID.String(),  // ← WRONG: this is not the DB row ID
    ...
}
```

**Impact:** The client calls `POST /tasks/:id/attachments/{attachmentID}/confirm`. The handler calls `repo.MarkComplete(attachmentID, userID)`. The DB has no row with that UUID — it has a different UUID generated by `gen_random_uuid()`. The query matches zero rows, returns `ErrNotFound`, and the service returns `422 ATTACHMENT_NOT_PENDING`. **The confirm flow is completely broken.**

**Fix — two options:**

**Option A (preferred — minimal change):** Use the UUID returned from `repo.Create()`:
```go
// Change:
_, err = s.repo.Create(ctx, ...)
// To:
attachment, err := s.repo.Create(ctx, taskID, userID, s3Key, req.MimeType, req.Filename, &sz)
if err != nil { ... }

return &PreRegisterResponse{
    AttachmentID: attachment.ID.String(),   // ← DB-generated UUID (matches future /confirm lookups)
    UploadURL:    uploadURL,
    ExpiresAt:    expiresAt,
}, nil
```

> Note: The S3 key will still contain the service-generated UUID — this is fine, because the server always looks up S3 keys by reading `s3_key` from the DB row, never by reconstructing it from the attachment ID.

**Option B (cleaner, slightly more invasive):** Pass the UUID to `repo.Create()` so the DB uses the same value as the service:
- Modify `Create()` signature to accept an `id uuid.UUID` parameter
- Use `INSERT INTO task_attachments (id, task_id, ...)` in the SQL
- Then use the same `attachmentID` for both S3 key and response

**Recommendation:** Option A. One-line change; no interface change required.

### Finding #2 — MINOR: `attachment_count` still absent from `TaskResponse` (NON-BLOCKING)

**Context:** AUD-002-BE Finding #2 / B-057 addressed the sort fix. The `attachment_count` field from CON-002 §3 was deferred from AUD-002-BE. Now that `task_attachments` table exists (migration 0008), this can be implemented.

**Required change:** In `task_repository.List()`, add a correlated subquery:
```sql
(SELECT COUNT(*) FROM task_attachments
 WHERE task_id = t.id AND status = 'COMPLETE') AS attachment_count
```
And add `AttachmentCount int` to `TaskResponse`.

**Verdict:** NON-BLOCKING. Can be filed as B-059 chore for the re-submit commit or the next available sprint touch.
