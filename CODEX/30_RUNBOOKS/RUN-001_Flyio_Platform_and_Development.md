# RUN-001 — Fly.io: Platform Overview & Local Development Guide

---

## 1. What Is Fly.io?

Fly.io is a platform-as-a-service (PaaS) that runs Docker containers on hardware distributed globally at the network edge. Unlike Heroku (which uses a centralized region), Fly.io runs your app close to your users via **Anycast routing** — traffic is automatically directed to the nearest healthy machine.

**Mental model:** You give Fly.io a `Dockerfile`. It builds it, stores the image, and runs it on their global infrastructure. You never SSH into a VM or configure Nginx. Fly handles TLS, load balancing, health checks, and restarts.

**Key facts for Task Nibbles:**
- The Go API binary is packaged in a Docker container and deployed to Fly.io
- The PostgreSQL database is a **Fly Postgres** cluster (Fly-managed, runs on their infrastructure)
- AWS S3 is external — deployed separately, accessed via SDK from within the Go container

---

## 2. Core Concepts

### 2.1 Apps
A Fly **App** is the top-level unit. Think of it as "your service." For Task Nibbles, we will have:
- `task-nibbles-api` — the Go backend
- `task-nibbles-db` — Fly Postgres cluster (created separately via `fly postgres create`)

Each app has a unique name across all of Fly.io. The app name becomes part of the default hostname: `task-nibbles-api.fly.dev`.

### 2.2 Machines
A Fly **Machine** is the underlying compute unit — a micro-VM that runs your container. Machines are fast to start (~300ms), cheap, and can **auto-stop** when idle (important for cost control on the free tier).

For an API, you typically run 2 machines minimum (for zero-downtime deploys). Fly handles this automatically.

### 2.3 Regions
Fly has data centers worldwide (Atlanta, Dallas, London, Singapore, etc.). You pick a **primary region** in `fly.toml`. For Task Nibbles (targeting US users initially), use `iad` (Ashburn, VA).

### 2.4 Volumes
Fly **Volumes** are persistent block storage — like an EBS volume. We do **not** use volumes for Task Nibbles data (Postgres handles persistence; S3 handles files). Volumes are noted here for completeness.

### 2.5 Anycast Networking
When a Flutter app makes a request to `api.tasknibbles.com`, Fly's Anycast routing sends it to the nearest healthy machine. Zero config required — this is automatic.

## 3. `fly.toml` — The Configuration File

`fly.toml` lives in the root of the `/backend` directory. It is the single source of truth for how Fly runs your app.

**Annotated example for Task Nibbles:**

```toml
# fly.toml — Task Nibbles API
# Generated by: fly launch
# DO NOT commit secrets here. Use: fly secrets set KEY=value

app = "task-nibbles-api"
primary_region = "iad"           # Ashburn, VA — closest to US East

[build]
  # Fly builds the Dockerfile in /backend at deploy time
  dockerfile = "Dockerfile"

[env]
  # Non-secret environment variables go here
  PORT = "8080"
  APP_ENV = "production"
  LOG_LEVEL = "info"
  # DATABASE_URL is set via: fly secrets set DATABASE_URL=...
  # AWS_ACCESS_KEY_ID is set via: fly secrets set AWS_ACCESS_KEY_ID=...

[http_service]
  internal_port = 8080           # Port your Go server listens on
  force_https = true             # Fly handles TLS termination
  auto_stop_machines = true      # Stop idle machines to save cost
  auto_start_machines = true     # Auto-start on incoming request
  min_machines_running = 0       # 0 = full idle (free tier); set to 1 for production

  [http_service.concurrency]
    type = "requests"
    hard_limit = 200             # Max concurrent requests per machine
    soft_limit = 150

[[vm]]
  size = "shared-cpu-1x"        # 256MB RAM — sufficient for Go API at MVP scale
  memory = "256mb"

[checks]
  [checks.health]
    grace_period = "10s"         # Wait before checking after deploy
    interval = "30s"
    method = "GET"
    path = "/health"             # Your Go API must expose GET /health → 200 OK
    port = 8080
    timeout = "5s"
    type = "http"
```

**Key rules:**
- ✅ Non-sensitive config (ports, log levels) → go in `[env]` in `fly.toml`
- ❌ Secrets (DB password, JWT secret, AWS keys) → NEVER in `fly.toml` → use `fly secrets set`

## 4. Secrets Management

Fly.io has a built-in secrets store. Secrets are encrypted at rest, injected as environment variables into your container at runtime, and **never visible in logs or the dashboard**.

### Setting secrets

```bash
# Set individual secrets
fly secrets set DATABASE_URL="postgres://user:pass@host:5432/task_nibbles?sslmode=require" \
              JWT_SECRET="your-256-bit-secret" \
              JWT_REFRESH_SECRET="another-256-bit-secret" \
              AWS_ACCESS_KEY_ID="AKIA..." \
              AWS_SECRET_ACCESS_KEY="..." \
              AWS_S3_BUCKET="task-nibbles-attachments" \
              AWS_REGION="us-east-1" \
              --app task-nibbles-api
```

```bash
# List secret names (values are never shown)
fly secrets list --app task-nibbles-api

# Remove a secret
fly secrets unset OLD_SECRET --app task-nibbles-api
```

### In your Go code

```go
// config/config.go
// Secrets are just environment variables — read them normally
type Config struct {
    DatabaseURL          string
    JWTSecret            string
    JWTRefreshSecret     string
    AWSAccessKeyID       string
    AWSSecretAccessKey   string
    AWSS3Bucket          string
    AWSRegion            string
    Port                 string
    AppEnv               string
}

func Load() (*Config, error) {
    cfg := &Config{
        DatabaseURL:      requireEnv("DATABASE_URL"),
        JWTSecret:        requireEnv("JWT_SECRET"),
        // ...
        Port:             getEnvOrDefault("PORT", "8080"),
    }
    return cfg, nil
}

func requireEnv(key string) string {
    v := os.Getenv(key)
    if v == "" {
        log.Fatalf("required environment variable %s is not set", key)
    }
    return v
}
```

> [!CAUTION]
> Never put secrets in `fly.toml`, `.env` files committed to git, or log statements. Use `fly secrets set` exclusively for sensitive values.

## 5. Local Development Workflow

### 5.1 Install the Fly CLI

```bash
# macOS
brew install flyctl

# Linux
curl -L https://fly.io/install.sh | sh

# Verify
fly version
```

### 5.2 Authenticate

```bash
fly auth login           # Opens browser → sign in → token stored locally
fly auth whoami          # Verify you're logged in
```

### 5.3 Connect to the Remote Postgres Database Locally

During development, you need to connect your local Go server to the Fly Postgres database (or a local Postgres instance). The recommended pattern:

**Option A — Local Postgres (fastest for day-to-day dev):**
```bash
# Run Postgres locally via Docker
docker run -d \
  --name task-nibbles-dev-db \
  -e POSTGRES_DB=task_nibbles \
  -e POSTGRES_USER=dev \
  -e POSTGRES_PASSWORD=devpassword \
  -p 5432:5432 \
  postgres:16-alpine

# Set DATABASE_URL in your local .env (NOT committed to git)
export DATABASE_URL="postgres://dev:devpassword@localhost:5432/task_nibbles?sslmode=disable"
```

**Option B — Proxy to Fly Postgres (use for integration testing):**
```bash
# Opens a WireGuard tunnel to the Fly Postgres cluster
# Forwards localhost:5433 → Fly DB
fly proxy 5433:5432 --app task-nibbles-db

# Then connect
export DATABASE_URL="postgres://postgres:PASSWORD@localhost:5433/task_nibbles?sslmode=disable"
```

### 5.4 Running the Go Server Locally

```bash
cd backend/

# Copy the example env file (add to .gitignore)
cp .env.example .env
# Edit .env with your local values

# Run with Air (hot reload)
go install github.com/air-verse/air@latest
air

# Or plain go run
go run ./cmd/api/main.go
```

### 5.5 `fly.toml` Local Override (optional)

For local testing with the Fly CLI tooling:
```bash
fly dev          # Runs your app locally through fly's local runner (if configured)
```
For Task Nibbles, plain `go run` or `air` is sufficient for local development.

## 6. PostgreSQL on Fly.io

### Provisioning (one-time, done by Architect/OPS sprint)

```bash
# Create the Fly Postgres cluster
fly postgres create \
  --name task-nibbles-db \
  --region iad \
  --initial-cluster-size 1 \       # 1 node = development; 2+ = production HA
  --vm-size shared-cpu-1x \
  --volume-size 10                 # 10 GB disk

# This outputs a connection string — save it immediately, store via fly secrets set
```

### Connecting

```bash
# Interactive psql session (for debugging, schema inspection)
fly postgres connect --app task-nibbles-db

# Run a one-off SQL command
fly postgres connect --app task-nibbles-db --command "SELECT version();"
```

### Connection String Format

```
postgres://postgres:PASSWORD@task-nibbles-db.internal:5432/task_nibbles?sslmode=require
```

The `.internal` hostname only works from within Fly's private network (i.e., from your deployed API container). This is how the API connects to the DB in production — zero public internet exposure.

## 7. Docker + Go — The Deployment Container

Fly deploys your app by building your `Dockerfile`. Go is ideal for Docker because it compiles to a single static binary with zero runtime dependencies.

**Required: Multi-stage Dockerfile (`/backend/Dockerfile`)**

```dockerfile
# Stage 1: Build
FROM golang:1.22-alpine AS builder

WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download

COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o /app/bin/api ./cmd/api/main.go

# Stage 2: Run
# Distroless = no shell, no package manager, minimal attack surface
FROM gcr.io/distroless/static-debian12

WORKDIR /app
COPY --from=builder /app/bin/api ./api

EXPOSE 8080
ENTRYPOINT ["/app/api"]
```

**Why multi-stage?**
- Build stage uses the full Go toolchain (~800 MB image)
- Final image is ~10 MB (just the binary + SSL certs)
- Dramatically faster deploy times, smaller attack surface

## 8. Logging and Observability

### View live logs

```bash
# Tail logs for your app
fly logs --app task-nibbles-api

# Filter to a specific region
fly logs --app task-nibbles-api --region iad
```

### From your Go code

Follow GOV-006 (Logging Specification). Use structured JSON logging — Fly's log aggregator parses JSON automatically, making logs searchable in the Fly dashboard.

```go
// Use slog (Go 1.21+ stdlib) or zerolog
import "log/slog"

slog.Info("task created",
    "task_id", task.ID,
    "user_id", userID,
    "type", task.Type,
)
```

### Metrics

Fly provides basic CPU/memory/request metrics at `fly.io/apps/task-nibbles-api/metrics`. No additional instrumentation required for MVP.

## 9. Cost Model (MVP)

| Resource | Config | Estimated Cost |
|:---------|:-------|:---------------|
| API Machine (shared-cpu-1x, 256MB) | auto-stop when idle | ~$0–$5/mo |
| Fly Postgres (shared-cpu-1x, 10 GB) | 1 node, always on | ~$0–$3/mo |
| Egress (API responses) | Low traffic initially | ~$0 |
| **Total MVP** | | **~$0–$8/mo** |

**Auto-stop behaviour:** When `auto_stop_machines = true` and `min_machines_running = 0`, Fly stops your machine after ~5 minutes of no traffic. On the next request, it cold-starts in ~300ms. For a personal-use MVP, this is perfectly acceptable.

For production with real users, set `min_machines_running = 1` to eliminate cold starts (~$5/mo).
